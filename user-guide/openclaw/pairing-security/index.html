3.10.6 配对与安全

OpenClaw 的网关不是默认对外开放的——首次连接需要配对认证，之后用配对生成的长效 token 验证身份。这一机制防止未授权的客户端连接板端 OpenClaw。

首次配对流程​

  1. PC 端 Studio 第一次访问板端 OpenClaw 时，OpenClaw 生成一次性 token
  2. token 通过 SSH 通道安全传递给 PC
  3. Studio 用该 token 完成配对，OpenClaw 记录该客户端为已授权
  4. 之后 Studio 用配对后的长效 token 与 OpenClaw 通信

整个流程对开发者透明，不需要手动操作。

配对管理​

OpenClaw tab 的 配对 子页签可以管理已配对的客户端：

操作说明
查看已配对客户端显示客户端 ID、配对时间、最后活跃时间
撤销配对移除某个客户端的授权，下次该客户端访问需要重新配对
查看待审批请求如果开启"审批模式"，新配对请求会出现在待审批列表

默认仅监听 127.0.0.1​

OpenClaw 网关默认仅绑定板端的 127.0.0.1:18789，不接受外部连接。这意味着：

  • 公网或局域网中的其他设备无法直接访问板端 OpenClaw
  • 只有通过 SSH 进入板端的客户端才能访问 OpenClaw（包括 Studio 用 SSH 隧道转发）
  • 即使板端 IP 暴露在公网，OpenClaw 端口本身也不可达

工具调用的白名单​

OpenClaw 的工具调用经过白名单过滤：

  • 默认允许的工具：远程命令执行、文件读写、网络访问、技能调用
  • 高风险工具（如系统级配置修改）需要在 OpenClaw 配置中显式启用
  • 危险命令（如 rm -rf、dd 等）触发时会要求用户在通道内二次确认

这一设计让 OpenClaw 可以放心暴露给外部通道（如飞书 Bot）使用，避免恶意指令直接破坏板端系统。

公网暴露的风险与建议​

做法风险
把 OpenClaw 端口 18789 直接暴露公网高（任何人可尝试连接）
修改默认绑定为 0.0.0.0高（绕过本地隔离）
把 SSH 端口暴露公网，依赖 SSH 隧道访问 OpenClaw中（依赖 SSH 安全性，需要密钥认证 + fail2ban）
通过 VPN 接入板端，不暴露任何端口到公网低（推荐做法）

生产环境建议：

  • 始终通过 SSH 隧道访问 OpenClaw，不要让 18789 端口直接对外
  • SSH 使用密钥认证，禁用密码登录
  • 板端配置防火墙，只开放必要的端口
  • 定期审计 OpenClaw 配对客户端列表，撤销不再使用的客户端
