# 3.13.3 安全策略与审批

外部通道（飞书、微信）默认应用比 Studio 内部更严格的审批策略。本节描述这些策略与最佳实践。

## 高风险命令的二次确认

通过外部通道触发的高风险操作会要求用户在通道内二次确认（不在 Studio 弹窗）：

| 触发命令 | 行为 |
|---|---|
| `rm -rf` | Bot 回复："即将删除 X，是否确认？请回复 /yes 继续。" |
| `kill` 关键进程 | 同上 |
| `dd` 磁盘写入 | 同上 |
| 修改 systemd 服务配置 | 同上 |
| 删除大量文件 | 同上 |

用户在通道内回复 `/yes`（飞书）或 `确认`（微信）后才执行；其他回复或超时（默认 60 秒）则取消。

这一机制防止"内网搞事的人 + 外网 Bot 接入"组合带来的安全风险——即使 Bot 被滥用，破坏性命令也无法直接执行。

## 多通道审批联动

| 触发方式 | 审批要求 |
|---|---|
| Studio 桌面客户端 | 由 risk 字段决定（low 自动、medium 提示、high 必确认） |
| 外部通道（feishu / weixin） | 高风险命令一律要求通道内二次确认，无论 risk 字段如何 |
| autonomy（OpenClaw 自主任务） | 仅执行预先在 OpenClaw 配置中允许的操作 |

## 安全最佳实践

| 实践 | 解释 |
|---|---|
| 生产环境用白名单或审批模式 | 不要开放任意飞书用户连 Bot |
| App Secret 不要提交 Git | 即使是私有仓库也不要——一旦仓库泄露代价高昂 |
| 高风险技能在多通道场景默认拒绝 | 必要时给特定白名单用户单独授权 |
| 定期审计配对客户端 | 移除不再使用或已离职同事的客户端 |
| 监控 Bot 调用日志 | 异常的高频调用可能是滥用或被入侵 |

## 关闭通道

不再使用某个通道时：

| 通道 | 关闭方式 |
|---|---|
| 飞书 | *配置中心 → 多通道 → 飞书 → 通道开关* 关掉 |
| 微信 | *配置中心 → 多通道 → 微信 → 通道控制 → 停止* |

关闭不会删除已配置的凭据与用户列表，重新启用时直接生效。如果完全不再使用，建议同时清理凭据避免误用。

## 应急响应

如果发现 Bot 被滥用或行为异常：

1. **立即停止通道**：在 *配置中心* 中关闭对应通道开关
2. **撤销凭据**：在飞书开发者后台或微信中撤销 App Secret 或解除绑定
3. **审计日志**：在 OpenClaw 的 *日志* 子页签中查看异常时间段的所有调用
4. **清理数据**：检查板端是否有异常文件或被修改的配置
5. **报告团队**：通知团队中所有有权限使用该 Bot 的成员

定期演练应急响应流程，确保事件发生时能快速反应。
