# 3.10.1 概述与适用场景

OpenClaw 不是 RDK Studio 的必需组件。本节帮助开发者判断"什么时候需要安装 OpenClaw、什么时候不需要"，避免不必要的部署成本。

## D-Moss 与 OpenClaw 的角色分工

| 角色 | 部署位置 | 擅长 |
|---|---|---|
| D-Moss Agent | PC 端 Studio 进程内 | 强模型推理、跨设备规划、长上下文、调用 PC 端工具与板端工具 |
| OpenClaw Agent | RDK 板端 systemd 长期服务 | 板端常驻、断网自治、紧贴硬件、长时间监控任务 |

两者都是完整的 AI Agent 运行时，差别在于部署位置与擅长的任务类型。

## 适用场景判定

| 场景 | 是否需要 OpenClaw |
|---|---|
| 偶尔通过 SSH 跑命令 | 不需要，远程终端就够 |
| 让 AI 帮跑命令（PC 在线） | 不需要，AI 通过 SSH 直接 `device_exec` |
| 板端需要常驻 Agent，PC 关机也要继续 | 必需 |
| PC 让板自己规划任务 | 必需 |
| 多板协作、跨板编排 | 必需（每个板都装 OpenClaw） |
| 微信、飞书 7×24 接入 | 推荐部署在板端，避免 PC 关机时通道中断 |

简单说：**临时调试不需要，常驻、自治、多板协同需要。**

## 典型使用场景

### 场景 A：板端长期监控

> "在板上每 5 分钟检查一次 BPU 温度，超过 70°C 自动降频。"

D-Moss 评估这是长期监控任务，委派给板端 OpenClaw。OpenClaw 在状态机中持续执行，PC 关机也不影响。当温度触发阈值时，OpenClaw 通过 SSH 隧道反向通知 PC（PC 在线时实时通知）。

### 场景 B：板端 Bot 7×24 在线

> "团队微信群里 @机器人查板端状态，即使我下班了也要能用。"

把微信通道部署在板端 OpenClaw，由板端处理所有微信消息。板端只要有电、有网就能工作，不依赖任何 PC。

### 场景 C：自动恢复

> "板端的关键服务挂了，我希望它自己重启。"

板端 OpenClaw 由 systemd 管理，自身崩溃会自动重启；OpenClaw 内的任务通过状态机持久化，重启后从最近 checkpoint 续跑。这种自愈能力让板端可以无人值守。

## 不适合 OpenClaw 的场景

| 场景 | 原因 |
|---|---|
| 板端硬件资源紧张（RAM &lt; 1 GB） | OpenClaw 本身占用 ~150 MB 内存 |
| 板端镜像不带 Node.js 22+ | OpenClaw 依赖较新版本 Node 运行时 |
| 单次任务，无需常驻 | 直接用 D-Moss + 远程终端更简单 |
| 板端无法访问外部网络 | 安装时需要从 npm 拉包 |

如果你的场景是"短期调试 + PC 始终在线"，可以完全不安装 OpenClaw，使用 D-Moss + 远程终端即可。
