# 3.3.2 断线重连机制

远程终端在网络波动或设备短暂不可达时不会立即关闭会话，而是尝试自动重连。这一机制让开发者在长会话中不需要担心"网络抖一下就要重新登录"。

## 重连触发条件

| 场景 | Studio 行为 |
|---|---|
| 网络抖动（短暂丢包） | 自动重连，已输入未发送的内容保留 |
| 设备短暂断电后恢复 | 显示"连接中断"横幅，恢复后自动重连 |
| 长时间不操作（&gt; 30 分钟） | 后台心跳维持，不主动断开 |
| 主动关闭标签 | 立即断开，不重连 |

## 重连过程中的会话状态

重连期间：

- 终端顶部出现黄色"连接中断，正在重连..."横幅
- 输入框可以继续输入，输入的内容暂存在前端，不会丢失
- 不接收板端的新输出（板端如果在产生输出，缓冲区可能丢失部分）

重连成功后：

- 黄色横幅消失
- 暂存的输入可以正常发送
- 板端会话从断开点继续——但已发送但未确认的命令的输出可能不会回放

## 心跳维持

为避免 SSH 会话因长时间无输入被防火墙或路由器断开，Studio 会每 30 秒发送一次空数据包维持连接。这种心跳对开发者透明，仅在防火墙策略严格的环境中可能需要调整（联系运维放行 SSH 长连接）。

## 重连仍失败

如果 Studio 多次重连失败（连续 5 次），会停止重连并提示开发者手动操作：

| 提示 | 建议操作 |
|---|---|
| `Connection refused` | 板端 sshd 服务可能停止，通过其他方式（如串口）登录板端检查 |
| `Connection timed out` | 网络完全不可达，确认板端电源与网络连接 |
| `Host key changed` | 板端可能重刷镜像，运行 `ssh-keygen -R <IP>` 清旧密钥后重连 |

完整的故障排查见 [5.2 SSH 连接失败](../../5-faq/2-ssh-failed.md)。
