# 3.9.3 在线状态监控

Studio 后台对每台已添加的设备进行心跳探测，及时反映设备的在线状态。开发者无需手动查询，列表上的状态指示器会持续更新。

## 心跳策略

| 阶段 | 探测频率 | 触发条件 |
|---|---|---|
| 设备在线、活跃使用中 | 每 30 秒一次 | 默认 |
| 连续 3 次心跳失败 | 标记为离线 | 网络抖动或板端临时不可达 |
| 离线后的低频探测 | 每 5 分钟一次 | 减少对已知离线设备的网络请求 |
| 探测成功 | 立即恢复在线标记 + 触发通知 | 设备恢复后自动感知 |

这种"在线高频、离线低频、恢复立即感知"的策略平衡了实时性与系统资源消耗。

## 在线但操作无响应的处理

有时心跳显示在线，但具体的命令操作（如远程终端中执行命令）卡住无响应。可能的原因：

- 心跳走的是简单的 SSH 连接探测，不依赖板端业务进程
- 而具体命令可能因为板端 CPU 高占用、IO 阻塞等原因卡住
- 这种情况通常需要在远程终端中验证：`uptime`、`top`、`iostat` 等

如果远程终端命令也无响应，可以通过 [3.7.4 RDK S100 xburn](../7-system-flashing/4-s100-xburn.md) 等其他方式或物理重启板端。

## 同设备并发限制

Studio 给同一台设备的 SSH 连接并发数设有上限（默认 8 路）。这个上限是为了避免长任务挤占所有 SSH 通道、阻塞所有探测。

在以下场景可能遇到并发受限：

- 同时打开多个远程终端标签 + AI Agent 同时调用 device_exec 工具 + 工作台心跳探测
- 文件管理批量上传大文件 + AI Agent 同时读取大量小文件

并发达到上限时，新的请求会进入等待队列，不会失败。这种排队是预期行为，不是错误。如果需要调整上限，可以通过环境变量 `RDK_DEVICE_EXEC_MAX_CONCURRENT` 修改（默认值 8）。

## 多人协作时的注意

如果一台设备被多个开发者通过 Studio 同时操作，可能出现：

- 一个开发者的命令影响另一个开发者的工作（例如 kill 了对方的进程）
- 设备列表里的 SSH 凭据被某人改了，其他人的 Studio 失去访问

建议团队约定：

- 生产设备指定专人值守，避免互相 kill 进程
- 关键设备使用 SSH 密钥认证，每个人有自己的密钥
- 在设备备注中标注"专人值守 / 共享调试"，让团队清楚使用规则
